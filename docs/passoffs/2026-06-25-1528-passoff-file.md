## Key decisions

- Shipped the kickoff gate even though it does not fix accuracy: it is a correct
  defensive measure (never show or count a game before it kicks off) and is
  source-agnostic, so it stays useful after the source is replaced.
- Did NOT pick a replacement data source unilaterally. Real live-football APIs
  need an API key and usually lack browser CORS, which collides with the app's
  "static site, no backend, no auth" design. That trade-off is the owner's call.

## Risks and open questions

- **The app shows fictional results.** `worldcup26.ir` is a simulation; its
  scores do not match the real 2026 World Cup (Group E feed 6-6-4-1 vs real
  7-6-4-0). Every standing, clinch lock, and projection downstream is only as
  accurate as this feed. This is the headline open issue.
- **Source replacement is unresolved.** The owner was asked how to proceed and
  selected "Something else" but has not yet named a specific feed/endpoint or
  approach. Three viable paths were laid out: (a) research authoritative APIs
  first; (b) real keyed API behind a thin serverless proxy (breaks "no
  backend"); (c) client-side key, no backend (key exposed, rate-limited). A
  fourth option: drop the live feed and maintain the (small) tournament results
  in the repo data island manually or via an update script.
- Whatever source is chosen, the name-map (`NAME_MAP`), the `(group, unordered
  name pair)` fixture match, and the offline snapshot all assume the current
  feed's shape and will need re-fitting.

## Lessons learned

Spent most of the session fixing *how* the app read the feed (refresh timing,
toggle state, kickoff gating) before checking *whether the feed is authoritative*.
The data was internally consistent and matched the feed perfectly, which masked
that the feed itself was fiction. Lesson: when a user reports "wrong data,"
validate the source's authority against ground truth early, before debugging the
consumption path.
